這裡的 Subquery 用了 OR：

and disable_flag ='N' or disable_flag is Null

因為 OR 的運算順序比 AND 低，結果 WHERE Clause 會變成：

where
(
 using_organization_id in (-1,0)
 and asl_status_id <> 5
 and item_id = b.inventory_item_id
 and disable_flag ='N'
)
or
( disable_flag is Null )

所以只要條件「disable_flag is Null」是 TRUE ，
那麼 EXISTS 也會是 TRUE 了。

我猜您其實想這樣寫的：

and ( disable_flag ='N' or disable_flag is Null )


另外，可否說說您原本想要做的功能或想撈的資料？（如果方便的話）
通常寫這一類用在 ERP 的 SQL ，最好參考 Oracle 的官方資料，例如
Oracle Report 檔案，因為這些 SQL 很複雜，很容易會撈錯資料。

補充一點，其實 EXISTS 與 IN 是兩個很不同的東西，
所以一般來說也不會用 EXISTS 取代 IN 。

就算真的可以取代，也未必會改善 Performance ，
我之前用類似的 SQL 在我的環境測試，結果也是 IN 比
較快。而且單看這裡兩個 SQL 來說，我也會猜用 IN
比較快，因為在前者的 Subquery，它的值是固定的，
但後者用了 EXISTS ，而 EXISTS 的運作原理就是：
「對於 Main Query 的每一行，都去執行一次 Subquery」
所以如果 mtl_system_items_b 有三千行，那麼 Subquery 也
會被執行三千次了。

就算真的是 Performance 較好，也未必會採用它，因為準確性
才是最重要的， Performance 還是次要。而且改善 Performance
也會帶來風險的，搞不好的話，輕則算錯數，重則整個軟件癱瘓，
這就是為何很多人會寧願用硬件去改善效能，而不去修改軟件了。

引本站一篇討論：
http://www.programmer-club.com/pc2020v5/forum/showsametitle.asp?board_pc2020=exp&id=6391

文中提到一家公司的電腦系統一直算不準庫存量，結果賠了幾百萬，
可以看到準確性有多重要。

記得有一個例子...

使用 IN --
Select city, Sum(bonus) From table2 B
Where yymm = '200401'
And key_no In (Select key_no From table1
  Where product_code='E')
Group By city;

使用 EXISTS --
Select city, Sum(bonus) From table2 B
Where yymm = '200401'
And Exists (Select * From table1 Where
  key_no = B.key_no And product_code='E')
Group By city;

說明：
1.因table2無product_code, 需經由table1判斷, 兩者關聯為key_no.
2.table1和table2為 Master-Detail 的關係.
3.Exists只需判斷資料是否存在, 故只要Select * 而不用指定特定欄位.
4.經測試第2個SQL比第1個平均快了6倍以上.

剛剛在新聞組找到一個關於 EXISTS 和 IN 的討論：
http://groups.google.com/groups?hl=en&lr=&ie=UTF-8&threadm=e2683c0b.0210080402.b4bab66%40posting.google.com

不知對這個情況是否適用，意思大概是這樣（如有譯錯請指正）：

<翻譯>

如果 table1.key_no 有 UNIQUE Constraint，那麼兩者會一樣快，
因為 IN 是需要做 DISTINCT 的步驟，如果有了 UNIQUE Constraint 就不用做了。

如果 table1.key_no 沒有 UNIQUE Constraint 而 table1.key_no 是有 Non-unique Index，
那麼因為 IN 不能使用這個 Index，所以需要做 DISTINCT 步驟，
但 EXISTS 可以用，所以 EXISTS 會比 IN 較快。

該文作者不建議永遠把 EXISTS 轉成 IN ，是因為如果那個 Subquery 只傳回很少行，
而且與 Main Query 是不相關的（沒有參照 Main Query），那麼 IN 比 EXISTS 快很多，
因為 IN 的 Subquery 只需執行一次（因為與 Main Query 不相關），但因為 EXISTS 必須
與 Main Query 相關，所以 Main Query 有多少行， Subquery 就要執行多少次了。

</翻譯>


我在工作中只有機會接觸 Oracle ERP 的 Report 檔案，而好像從來沒見過它用 EXISTS ，
IN 也很少見，全部都是用 Table Joins ，所以沒有機會看到 EXISTS 的應用。

>說明：
>1.因table2無product_code, 需經由table1判斷, 兩者關聯為key_no.
>2.table1和table2為 Master-Detail 的關係.
>3.Exists只需判斷資料是否存在, 故只要Select * 而不用指定特定欄位.
>4.經測試第2個SQL比第1個平均快了6倍以上.



剛剛測試過三種寫法： Table Join , IN 和 EXISTS ，結果是 EXISTS 慢一點點。

測試平台： Windows 2000 Pro , Personal Oracle 8.1.7 , 用 FREE TOAD 來試。

結果：

1) 當 IN 的 Subquery 傳回 10000 個 DISTINCT 的行：

a) 如果 KEY_NO 沒有 FK 和 PK：
Table Join = 1 秒
In = 1 秒
EXISTS = 1 分 46 秒

b) 如果有 FK 和 PK：
Table Join = 1 秒
In = 1 秒
EXISTS = 1 至 2 秒


1) 當 IN 的 Subquery 傳回 50000 個 DISTINCT 的行：

Table Join = 1 秒
In = 1 秒
EXISTS = 2 秒


結論：大概要看資料特性和外圍環境因素吧。
